# Skill Framework Master
分类：6. 技能框架、元管理和生态自省

本文件定义了本项目中所有 `.md` 技能文件的标准结构和编写原则。Grok 在读取项目中的其他技能文件时，应遵循本框架进行逻辑解析。  
**本框架支持版本演进**，所有重大变更均应在「版本演进」章节中记录。

## 1. 技能文件标准结构 (Standard Structure)

每个技能文件必须包含以下核心板块：

### # [技能名称] (Skill Name)
- **定义**：简洁、唯一的技能名称（例如：`Code_Reviewer` 或 `Market_Analyst`）。
- **目标**：用一句话描述该技能最终要解决什么问题。

### ## 角色设定 (Persona)
- **身份描述**：定义 Grok 在执行此技能时的专业背景、语气和思维方式。
- **专业水平**：明确其在该领域的资历（例如：Senior Engineer, Academic Researcher）。

### ## 触发场景 (Trigger Context)
- **识别信号**：描述哪些关键词或任务类型应激活此技能。
- **输入要求**：明确用户需要提供哪些信息才能有效触发。

### ## 核心工作流 (Core Workflow)
- **步骤拆解**：使用编号列表描述执行任务的具体逻辑步骤。
- **思维链 (Chain of Thought)**：要求 Grok 在输出最终结果前，先进行内部逻辑推理或步骤确认。

### ## 约束与红线 (Constraints & Guardrails)
- **禁止项**：明确绝对不能做的事情（例如：禁止编造数据、禁止使用过度口语化的表达）。
- **必须项**：必须遵循的规范（例如：必须引用来源、必须检查代码兼容性）。

### ## 输出规格 (Output Specification)
- **格式要求**：指定返回结果的结构（例如：Markdown 表格、JSON 块、带有二级标题的报告）。
- **语言风格**：定义语体（专业、简洁、启发式等）。

### ## 示例 (Few-shot Examples)（可选但推荐）
- 提供 1-2 个“输入-输出”对，帮助 Grok 理解执行该技能的最佳实践。

## 2. 编写原则 (Writing Principles)

1. **原子化 (Atomization)**：一个文件只处理一个独立技能，避免功能耦合。
2. **显性化 (Explicitness)**：不要假设 Grok 知道默认规则，所有关键逻辑必须写在纸面上。
3. **结构化 (Structured Prompting)**：尽可能使用标题、列表和代码块，减少大段叙述。
4. **动态占位符**：使用 `{{Input}}` 或 `[Variable]` 来提示 Grok 这里是需要替换的动态内容。

## 3. 技能分类层级规范（v0.5 – 2026-03-05）

现有技能应尽快补充该元数据行。分类用于辅助 Orchestrator 进行更精准的 tie-breaker 和 Fuzzy 路由。

**头部元数据推荐位置**：文件第二行，独占一行，无前导空格。

## 版本号升级策略 (Version Upgrade Policy)

**V1系列 → V2.0 切换标准**（满足任一条件即可启动全系统升级）：
1. **架构级重构**：Skill Framework Master 核心结构发生重大变更（新增核心板块、工作流范式彻底重写、分类体系调整超过5个大类、引入持久化记忆层或原生多代理协作等）。
2. **规模与迭代里程碑**：技能文件总数超过20个，且至少70%技能完成 v1.5+ 迭代；或全系统累计小版本迭代总数达到30个以上。
3. **审计与用户驱动**：Skill Auditor 全局审查后判定存在3个以上系统级问题需整体重构，或用户明确要求“进入V2.0”“重置版本号标志新阶段”并确认。
4. **功能里程碑**：引入全新跨技能基础能力（例如统一用户偏好记忆系统、AI自进化闭环、工具链深度原生集成）。

**执行方式**：
- 由 Self-Updating Skill Engineer 统一生成 V2.0 模板文件。
- 所有现有技能文件需追加迁移说明（在版本演进表中新增一行“V2.0 迁移”记录）。
- 新建技能默认从 v2.0.0 开始计数。
- 本政策默认适用于后续所有 .md 技能文件。

## 5. 版本历史管理规范 (Version History Management)（v1.6 – 2026-03-06）

所有技能文件必须遵守以下规则（取代之前所有版本历史方案）：

- 「版本演进」表格**永久只保留**最近 5 条记录 + 所有重大里程碑（v1.0、v2.0 等）。
- 超过 5 条的旧记录直接删除（不保留在文件中）。
- 想保留完整历史，可自行将旧版文件存档或使用外部工具（Git/Notion 等）。
- 此规范适用于**所有现有及新建技能文件**，现有文件立即按此精简。
